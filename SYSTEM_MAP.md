# SYSTEM_MAP — карта взаємозвʼязаної системи FINEKO

> Мета: продавати кожен продукт окремо, але тримати їх як одну систему, щоб потім
> захопити всю інфраструктуру компанії-клієнта. Цей документ фіксує сервіси,
> точки перетину, пробіли/нестиковки і порядок робіт. Оновлюється по ходу.
> Останнє оновлення: 2026-07-17.

## 2. Спільна інфраструктура (перетини)

- **FLOWS = хаб ботів і оркестрації.** Тут живуть: орг-агент (тонкий → `/agent/act` платформи), контент-scheduler, раннер-cron (#233), browser-MCP (#249), Telegram-самообслуговування (#227), воронка найму (#262). Принцип: **тонкий агент у flows → товстий API у продукті**.
- **Google Drive = сховище правди для документів:** інструкції (ОРГ), контент-доки, таблиці кандидатів (#262), папки бази знань.
- **Вектор-пошук** зараз вбудований у ОРГ (pgvector: 235 чанків знань + embedding інструкцій). #263 виносить його в окремий сервіс з токен-доступом до папок → спільний для ОРГ/КОНТЕНТ/FLOWS/ботів.
- **MCP** — у ТРЕКЕРА (Claude.ai конектор з OAuth), у КОНТЕНТ (#245 продукти), browser-MCP (#249). Патерн OAuth трекера — взірець для решти.
- **Ідентичність/компанії** дублюються: у ТРЕКЕРА свої акаунти+ролі, в ОРГ будуються свої (#235/#238). #253 хоче крос-імпорт.

## 4. Пробіли й нестиковки (потребують рішення / фундаменту)

- **G1. Єдина ідентичність (#281).** ТРЕКЕР і ОРГ будують окремі акаунти. Перед #235 обрати: (А) спільний SSO; (Б) один продукт — джерело правди; (В) поки окремо + синхр. через #253. **Рекомендація:** #235 будувати на моделі акаунтів трекера, щоб потім злити.
- **G2. Вектор-мікросервіс — прихований фундамент (#280→#263).** Від нього залежать і #224 (ОРГ-розповсюдження), і міграція контент-KB (#282). **Рекомендація:** підняти #263 у пріоритеті й зробити РАНІШЕ за залежні.
- **G3. Інструкції — фундамент (#276). РІШЕННЯ ВЛАСНИКА:** зміст інструкцій лишається
  на **Google Drive (Docs/Sheets)** — людям звично + дає структуру папок інструкцій на
  диску. У БД `Instruction` = *посилання* на Drive-файл + метадані + звʼязки за id
  (посада/процеси) + **індексація у вектор-мікросервісі**, щоб ОРГ швидко знаходила
  інструкцію для зміни. ОРГ має інструменти редагування, що пишуть у Drive-файл. Журнал
  (#222) фіксує **лише що змінилось** (підсумок), не весь текст. Розблоковує
  #222/#224/#226/#206/#212.
- **G4. Сценарій інтервʼю (#277). РІШЕННЯ ВЛАСНИКА:** агент живе **у flows-воронці**
  (як усі наші агенти) — щоб бачити історію запусків, запити до API і помилки. Воронка
  веде стадійний сценарій (INTERVIEW_PLAN.md) і викликає товстий API платформи для
  запису. Платформа — не тонкий релей, а виконавець дій + синхронізація (#279).
- **G5. Спільні сповіщення/SMTP.** #192 (пінги) і #237 (скид пароля) і #262 (таблиці) потребують email/Telegram-надсилання. **Рекомендація:** єдиний notifications-концерн (SMTP + Telegram-send через flows).
