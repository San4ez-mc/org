# INTERVIEW_PLAN — універсальний сценарій інтервʼю орг-агента (#277)

> Спец для агента, що будує орг.структуру + процеси + інструкції через діалог у
> Telegram (flows → `/agent/act`). Принцип: **від загального до вузького, зверху
> вниз**; кожна відповідь синхронно доповнює всі три сутності (див. #279).
> Агент веде цей стадійний план (state-machine), flows лишається тонким релеєм.

## Архітектура (рішення власника: агент живе у FLOWS)
Агент-інтервʼю — це **воронка у flows**, як і решта наших агентів. Причина: у flows
видно історію запусків, запити до API і помилки (спостережуваність).
- **flows-воронка** веде сам сценарій (стадії, питання, LLM-парсинг відповідей у
  структуровані дані) і на кожному кроці **викликає API платформи** для запису
  (створити відділення/відділ/посаду/процес, згенерувати інструкцію). Стан діалогу
  тримає flows (як і в інших воронках).
- **Платформа** — «товстий» API з гранулярними інструментами (більшість ендпоінтів
  уже є: `/companies/:id/org-units`, `/members`, `/processes`, імпорт #209 тощо) +
  `/agent/act` для складених інтентів. Платформа виконує запис у БД і запускає
  синхронні пропозиції (#279); RAG/вектор — через мікросервіс (#263).
- **LLM** (у ноді flows) використовує базу знань (Хаббард/Адізес) і накладає бізнес
  клієнта на канонічний борд (7 відділень), а не генерує з нуля.
- Спостережуваність: усі кроки, API-виклики й помилки — в історії запусків flows.

## Стадії (state-machine)

### Стадія 0 — Власник, бачення, стратегічний шар (опори: #213, #215)
Мета: зрозуміти навіщо існує компанія і хто нею володіє.
Питання (по одному, з уточненнями):
1. Чим займається компанія, який головний продукт/послуга?
2. Мета компанії та її **ЦКП** (цінний кінцевий продукт усієї компанії)?
3. Ідеальна картина через 1–3 роки?
4. Хто власники/інвестори і які долі (%)? (→ модель owners, #215)
5. Скільки зараз людей; хто ти в структурі (власник/директор)?
Будує: `Company` (мета, ЦКП), `owners`, верхівку борду (Власник → Викон.директор).

### Стадія 1 — 7 відділень (канонічний борд Хаббарда)
Агент пояснює 7 відділень і **накладає** бізнес клієнта на них (не вигадує):
адміністративне, побудови/персоналу, розповсюдження/маркетинг+продажі,
фінансове, технічне/виробництво, кваліфікації, по роботі з публікою.
Питання: які з цих напрямів уже є де-факто, хто за них відповідає?
Будує: `OrgUnit(type=DIVISION)` ×7 з `boardNo`, ЦКП кожного відділення, керівники.

### Стадія 2 — Відділи всередині відділень + ЦКП (опора: #216)
Для кожного відділення: які відділи/секції є (з канонічних 21 як підказка),
їх ЦКП, керівники.
Будує: `OrgUnit(type=DEPARTMENT/SECTION)`, ЦКП, `reportsTo`.

### Стадія 3 — Посади + ЦКП + люди
Для кожного відділу: які посади, ЦКП посади, хто обіймає (людина або вакансія).
Будує: `OrgUnit(type=POST)` з ЦКП, `Member` + `MemberPost`, `isVacant`.
Тут же можна масово залити наявних (#209 імпорт).

### Стадія 4 — Бізнес-процеси (потоки цінності)
Для головних потоків цінності: кроки (відповідальна **посада за id** #221 → дія →
результат), гілки-рішення, болі. Агент пропонує 2–4 ключові процеси.
Будує: `Process` (steps/graph), звʼязок кроку з посадою за id.

### Стадія 5 — Посадові інструкції (опора: #204, #276)
Для кожної посади генерує чернетку інструкції з ЦКП + кроків процесів, де посада бере
участь. Правила генерації — #204.
**Джерело правди — Google Drive (Docs/Sheets), бо людям звично і дає структуру папок
інструкцій на диску.** У БД `Instruction` (#276) — лише *посилання* на Drive-файл +
метадані + звʼязки за id (посада/процеси) + індексація у вектор-мікросервісі (#263),
щоб ОРГ швидко знаходила потрібну інструкцію для зміни. Редагування — інструментами
ОРГ, що пишуть у Drive-файл. У журнал (#222) пишемо **тільки що змінилось** (короткий
підсумок), не весь текст.

## Синхронність (движок #279) — тригери під час інтервʼю
- Додали **посаду** (Стадія 3) → запропонувати: її кроки у відповідних процесах +
  чернетку інструкції.
- Змінили **крок процесу** (Стадія 4) → оновити інструкцію відповідальної посади.
- Змінили **інструкцію** (Стадія 5) → вектор-пошук повʼязаних (#224, через
  мікросервіс #263) → запропонувати правки повʼязаних.
- Усі зміни йдуть як `Proposal` (#223) → людина підтверджує inline-кнопками.

## Правила діалогу
- Одне питання за раз; коротко; approve/skip кнопками де можливо.
- Дозволяти «не знаю / пропустити» — лишати одиницю чернеткою (DRAFT), позначати як
  прогалину у діагностиці (#210).
- Після кожної стадії — коротке резюме що збудовано + «далі?».
- Мова лише українська.
- Зберігати стан: перерваний діалог продовжується з тієї ж стадії.

## Крайові випадки
- Компанія вже має структуру → інтервʼю переходить у режим доповнення/аудиту (#208).
- Дуже мала компанія (<7 напрямів) → відділення можуть суміщатись однією людиною;
  борд усе одно з 7, але посади можуть перетинатись.
- Імпорт файлом (#209) як альтернатива Стадіям 2–3.
