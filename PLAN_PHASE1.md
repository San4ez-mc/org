# План Фази 1 (Sheets-first MVP)

> Мета Фази 1: студент проходить опитування в Telegram-боті → ШІ генерує орг.структуру,
> бізнес-процеси і посадові інструкції (чернетка) → усе лягає в Google Sheet (оргсхема) +
> Google Drive (папки та Docs-інструкції) → індексується у pgvector → через бот можна
> редагувати інструкції (з розповсюдженням через proposals) і завантажувати документи.
> Без веб-інтерфейсу.

---

## 1. Канонічна модель (хребет системи)

Обидва референси власника (шаблон папок + оргсхема) побудовані на **7-відділенній оргсхемі**.
Ця модель зашита в платформу як `canonical-org-template` і є каркасом, на який ШІ накладає бізнес клієнта.

### Ієрархія одиниць
```
Компанія
 └ Власник / Рада засновників
   └ Виконавчий директор
     └ Заступники (адмін., тех., по роботі з публікою)
       └ Відділення (Division) ×7
         └ Відділ (Department) — нумеровані 1..21
           └ Секція (Section)
             └ Посада (Post)
```

### 7 відділень (фіксований порядок борду 7-1-2-3-4-5-6)
| № | Відділення | Фокус |
|---|-----------|-------|
| 7 | Адміністративне | стратегія, керівництво, персонал верхнього рівня |
| 1 | Побудови (HR/Персонал) | найм, введення в посаду, посадові папки, оргсхема |
| 2 | Поширення (Маркетинг+Продажі) | просування, продажі |
| 3 | Фінансове | дохід, розхід, облік |
| 4 | Технічне | виробництво продукту/послуги |
| 5 | Кваліфікації | якість, навчання персоналу, вдосконалення |
| 6 | По роботі з публікою | PR, партнери, ввідні послуги |

### Ключові атрибути одиниці
- `name`, `type` (division/department/section/post), `parent`
- **ЦКП** (Цінний Кінцевий Продукт) — обов'язковий для кожної одиниці
- посада: `title`, `holder` (ПІБ або вакансія), `reports_to`

> ШІ не вигадує структуру з нуля — він **мапить** функції клієнта на цей борд, прибирає зайве,
> додає специфічні секції/посади, заповнює ЦКП. Малий бізнес (5 осіб) = багато посад суміщає одна людина.

## 2. Модель даних Фази 1 (Prisma + pgvector)

Мінімум, але одразу з правильними зв'язками (щоб Ф2 не переробляти):

- `companies` — `id, name, abbr, drive_root_folder_id, org_sheet_id, link_code, created_at`
- `org_units` — `id, company_id, parent_id, type(division|department|section|post), name, ckp(text), order_no, board_no` — оргсхема в БД (джерело зв'язків; Sheet — редагований дзеркало-вигляд)
- `posts` — розширення org_unit type=post: `holder_name, reports_to_unit_id, is_vacant`
- `members` — `id, company_id, telegram_user_id, full_name, role(owner|head|employee), post_unit_id`
- `bot_links` — `code, company_id, member_id?, role, expires_at` — deep-link підключення до спільного бота
- `processes` — `id, company_id, name, description, owner_unit_id, steps(json)`
- `instructions` — `id, company_id, post_unit_id, process_id?, title, drive_doc_id, folder_path, status(draft|active), version, updated_at`
- `instruction_links` — `instruction_id, related_instruction_id, relation_type, weight` — граф залежностей
- `instruction_embeddings` (pgvector) — `id, company_id, instruction_id, chunk_no, content, embedding, drive_doc_id, updated_at`
- `drive_folders` — `company_id, org_unit_id, path, drive_folder_id`
- `proposals` — `id, company_id, type(instruction_edit|new_doc|structure_change), target_instruction_id?, payload(json), status(pending|approved|rejected), created_by, approved_by, created_at`
- (слоти на майбутнє, поки не використовуються) `companies.subscription_plan/status/trial_ends_at`

## 3. Генерація Google Drive (з оргсхеми)

Для кожної `post`-одиниці створюється папка з 4 елементами (за шаблоном власника):
```
<Відділення>/<Відділ або функція>/Посадові папки/<Посада>/
  ├─ Посадові інструкції   (папка)   ← сюди Docs-інструкції
  ├─ Доступні документи    (папка)
  ├─ Доступи               (таблиця)
  └─ Звітність             (таблиця)
```
`drive_folders` мапить кожен `org_unit` на `drive_folder_id`. Корінь — папка компанії з `abbr`-префіксом.
Пакет `packages/drive` (порт логіки з ПСЗ на Node) інкапсулює створення тек/таблиць/Docs.

## 4. Онбординг-опитування (мердж воронки + fineko)

**UX діалогу** — з воронки курсу «Налаштування фін. звітності» (система воронок): покрокові
питання, збереження відповідей, чекпоінти.
**Генерація** — логіка fineko (`conversational-audit-flow` + `work-plan-flow`), адаптована під канонічну модель.

### Кроки
1. Бот питає про бізнес: сфера, продукт/послуга, кількість людей, ключові функції, хто що робить, поточні болі. (5–12 питань, адаптивно.)
2. `POST /api/onboarding/answer` — платформа зберігає відповіді.
3. `POST /api/onboarding/generate` — ШІ:
   - мапить функції на 7 відділень → відділи → секції → посади;
   - для кожної одиниці формулює **ЦКП**;
   - визначає, які посади суміщає одна людина (для 5 осіб);
   - генерує 3–7 ключових **бізнес-процесів**;
   - генерує **посадові інструкції** (DRAFT) для кожної посади.
4. Платформа: створює `org_units`, пише **оргсхему в Google Sheet**, створює **папки Drive**,
   створює **Docs-інструкції**, індексує все в pgvector.
5. Бот показує підсумок + посилання на Sheet і кореневу папку Drive. Усе — чернетка для правок.

---

## 5. Зміна інструкції + розповсюдження

1. Співробітник у боті: «зміни в інструкції рекрутера пункт про …».
2. `POST /api/instructions/:id/edit` → ШІ оновлює Doc, ре-ембедить.
3. Worker (Bull): вектор-пошук пов'язаних інструкцій (+`instruction_links`) → ШІ формує
   `proposals` (type=instruction_edit) для кожної зачепленої.
4. Відповідальний керівник отримує в Telegram картку пропозиції з inline-кнопками
   **Підтвердити / Відхилити** (доступ за роллю: голова відділення — свій відділ, власник — уся компанія).
5. На «Підтвердити» — Doc оновлюється, ре-ембединг, `instruction.version++`.

---

## 6. Завантаження документа через бот

1. Користувач кидає файл у бот.
2. `POST /api/documents/ingest` — витяг тексту (docx/pdf/txt), ембединг.
3. ШІ класифікує: якій посаді/процесу/теці належить (вектор-пошук по існуючих + LLM).
4. Створюється Google Doc у правильній теці Drive → `instructions` (або «Доступні документи») + `instruction_links` → індексація.
5. Бот підтверджує куди поклав; якщо непевність — питає уточнення.

---

## 7. Маршрутизація спільного бота

Один бот платформи. Підключення: `https://t.me/<bot>?start=<link_code>`.
`/start <code>` → резолв `bot_links` → компанія + роль → прив'язка `telegram_user_id` до `member`.
Далі всі дії — в контексті цієї компанії/ролі. Бот живе у воронках і б'є в API платформи.

## 7b. Уточнена модель структури Drive (узгоджено 2026-07-09)

**Принцип:** джерело правди — БД платформи; Drive — згенерована проекція. Люди керують через бота → БД → reconcile у Drive.

**Оригінали інструкцій — тільки у Відділенні побудови**, у папках посад — ярлики (перегляд):
```
<Компанія>/
├── 1. Відділення побудови/
│   ├── Посадові інструкції/  (ОРИГІНАЛИ, дзеркало оргсхеми)  ← єдине джерело
│   ├── Правила створення посадових інструкцій (Doc)
│   └── <відділи>/Посадові папки/<Посада>/ …
├── 2…7 відділення/<Відділ>/Посадові папки/<Посада>/
│   ├── ➜ ярлик на «Посадову інструкцію» (оригінал у Побудові)
│   ├── Доступні документи/ · Доступи(табл) · Звітність(табл)
├── Робочі папки/
│   ├── <Працівник>/ { Посади (ярлики на посади), Робоча папка (особисте) }
│   └── Архів/  (папки звільнених)
└── Оргсхема(табл) + Персонал(табл)   ← дашборд власника/HR
```
Один оригінал → багато ярликів = зміна оригіналу оновлюється у всіх тримачів посади (механізм «живого організму» на рівні Drive). Реалізовано в `packages/drive/buildCompany.ts`.

**Доступи = похідне від посади + винятки.** Тримаєш посаду → автоматично маєш доступ до її папок. Таблиця «Доступи» = згенерований вигляд + ручні винятки. Мeханізм «галочка→доступ» — **платформа звіряє** таблицю з правами Drive (reconcile-worker), без Apps Script. Апгрейд на Workspace → Google Groups на посаду.

**Сценарії (одна команда бота, мутує БД → reconcile):** додати працівника (Робоча папка+Посади+доступи), друга людина на ту ж посаду (спільні ярлики+оригінал), підвищення/переведення (та сама особиста папка, міняються ярлики+доступи), звільнення (папка→Архів, зняти всі доступи, посади→вакансія), тимчасова передача повноважень (призначення з датою закінчення, авто-зняття), вакансія (посада без тримача).

**Варіативність відділів:** канонічний 7-відділенний борд — каркас; відділи/секції/посади всередині різняться за типом бізнесу → ШІ мапить конкретний бізнес на борд (I4), потрібна бібліотека шаблонів під типи бізнесу (власник надасть приклади).

## 8. API платформи (для бота-воронки)

| Метод | Ендпойнт | Призначення |
|------|----------|-------------|
| POST | `/api/bot/link/resolve` | резолв deep-link коду → компанія+роль |
| POST | `/api/onboarding/answer` | зберегти відповідь опитування |
| POST | `/api/onboarding/generate` | згенерувати структуру+процеси+інструкції |
| GET  | `/api/companies/:id/summary` | підсумок (посилання на Sheet/Drive) |
| POST | `/api/instructions/:id/edit` | правка інструкції (запускає розповсюдження) |
| GET  | `/api/proposals?status=pending&scope=` | список пропозицій за роллю |
| POST | `/api/proposals/:id/approve` \| `/reject` | рішення по пропозиції |
| POST | `/api/documents/ingest` | завантажити документ через бот |
| POST | `/api/search` | семантичний пошук по інструкціях компанії |

Auth: Bearer-токен платформи (як MCP_SECRET у воронках) + `company_id` з `bot_links`.

## 9. Монорепо та стек

```
platform/
├── apps/
│   ├── api/            Express API (ендпойнти §8)
│   └── worker/         Bull: розповсюдження, ре-індексація, сповіщення
├── packages/
│   ├── db/             Prisma + pgvector (§2)
│   ├── ai/             Claude (генерація/класифікація) + ембединги
│   ├── drive/          Google Drive/Docs/Sheets (порт ПСЗ на Node)
│   ├── telegram/       сповіщення/картки пропозицій
│   └── org-template/   канонічна 7-відділенна модель (§1) + мапер
└── prisma/schema.prisma
```
Стек: Node.js + TypeScript, PostgreSQL+pgvector, Bull, Google APIs, Anthropic SDK.
Деплой: VPS + PM2 + git-pull (як воронки). Middleware/патерни — з fineko backend_new.

**Ембединги (рішення):** Vertex `text-embedding-004` (мультимовний, вже в ПСЗ, Google-екосистема),
fallback OpenAI `text-embedding-3-small`. Розмірність фіксуємо в pgvector-колонці.

## 10. Порядок реалізації (інкременти)

1. **I1 — Скелет + БД.** Монорепо, Prisma-схема §2, pgvector, підключення до Postgres.
2. **I2 — org-template + мапер.** Канонічна модель §1 як дані + функція мапінгу бізнесу на борд.
3. **I3 — packages/drive.** Створення папок/таблиць/Docs; запис оргсхеми в Sheet.
4. **I4 — packages/ai + генерація.** Промпти генерації структури/процесів/інструкцій.
5. **I5 — API онбордингу** (`answer`, `generate`, `summary`) + індексація в pgvector.
6. **I6 — Бот-воронка онбордингу** у системі воронок, що б'є в I5 (мердж UX воронки курсу).
7. **I7 — Правки + proposals + worker розповсюдження** (§5) + картки в Telegram.
8. **I8 — Завантаження документів через бот** (§6).
9. **I9 — Пілот** на реальній компанії власника; калібрування промптів.
