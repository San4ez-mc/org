# ДОКУМЕНТ 2. Правки до наявної Орг.Платформи FINEKO

> Шлях еволюції наявної системи (де майже все вже [Є]) до архітектури з Документа 1.
> Спирається на вашу візію (статуси [Є]/[Част]/[Нема]) + модель "консультант-диригент" +
> три ваші рішення (ієрархія / тихий канал / видимі конфлікти).

---

## 0. Головний висновок: майже все ядро вже є — бракує трьох речей

На відміну від контент-системи, тут переписувати з нуля НЕ треба взагалі. У вас уже [Є]:
борд 7 відділень, посади, ЦКП, PAEI, процеси з React Flow, зв'язок кроку з посадою за id,
стадійний онбординг-агент, вектор-пошук пов'язаного, дашборд власника, org-health аудит,
SSO, вартість ФОП, портфель клієнтів. Це 70%+ Документа 1.

**Бракує рівно трьох речей, і всі три — наслідок того, що систему проектували як SaaS для
власника, а не як інструмент диригента:**

1. **Немає провенансу й контуру довіри до даних** — система зберігає факт, але не "хто
   сказав, підтверджено, чи є розбіжність". Без цього ваша роль контролера технічно
   неможлива. Це головна прогалина.
2. **Немає Rule Engine + Event Bus як окремих шарів** — "живість" тримається на вектор-
   пропозиціях за запитом; фонового детермінованого сторожа й подієвої реакції нема
   (ваша візія, розділ 5, п.1-2 — самі це відзначили).
3. **Немає Delivery Engine** — методологія існує як стадії онбордингу, але не як машина
   станів з критеріями готовності, панеллю контролю й керуванням кількома впровадженнями.

Усе інше — доповнення до наявного, не перебудова.

---

## 1. Провенанс і контур довіри (найбільша й найважливіша правка)

### 1.1. Модель даних — додати до КОЖНОГО факту
Кожна сутність/поле, що прийшло від людини чи документа, отримує провенанс:
`source_type (owner|head|employee|document|agent)`, `source_id`, `stated_at`,
`confidence`, `status (proposed|approved|disputed|unconfirmed)`, `approved_by`,
`conflict_group_id?`. Це адитивна міграція, наявні зв'язки не ламаються.

### 1.2. Статус факту замість "просто записано"
Зараз факт або є, або нема. Стає: `proposed` (агент витяг, чекає) → `approved` (керівник
підтвердив) → або `disputed` (суперечить іншому джерелу) / `unconfirmed` (одне джерело,
не перевірено). Org-health і Delivery читають ці статуси.

### 1.3. Панель довіри (новий екран для вас)
Поверх наявного org-health аудиту: по компанії — % approved / unconfirmed / disputed,
список спірних фактів, список "не покрито" (чеклист повноти нижче). Це ваш робочий екран.

---

## 2. Rule Engine — виділити в окремий шар (частина логіки вже є в аудиті)

Ваш org-health аудит уже знаходить "прогалини, вакансії, недоописані процеси" — це
зародок Rule Engine. Треба:
- Винести перевірки в явний детермінований движок (без ШІ), розширити набором з Документа 1
  §2 (цикли підпорядкування, двоє за одне ЦКП, крок без відповідального, процес без метрики,
  відділення без керівника, перевантаження).
- Кожне правило повертає структурований факт "діра тут" з посиланням на вузол — для
  Delivery (критерії готовності) і панелі довіри.
- **Не на вектор.** Це реляційні запити по графу — швидко, дешево, детерміновано. Вектор
  лишається тільки для "знайди семантично схоже" (Шар 3).

---

## 3. Event Bus — додати шину подій (закриває "реальний час" з візії)

Зараз пропозиції — за запитом/подією без централізованої шини. Додати чергу подій (flows
це вже вміє — та сама механіка, що в контент-воронках):
- Публікатори: будь-яка зміна графа через API.
- Підписники: Rule Engine (перевір), Consistency-агент (знайди пов'язане), індексатор
  (онови вектор), нотифікації.
- Це перетворює "сторожа консистентності [Нема]" з вашої візії на фонового подієвого
  підписника — без окремого великого проєкту.
- **Ідемпотентність подій обов'язкова** (урок контент-системи: без неї — подвійні
  спрацювання; тут це були б подвійні пропозиції/конфлікти).

---

## 4. Три канали введення (з ваших рішень)

### 4.1. Ієрархічний (default) — доробити чергу затвердження
Дані співробітника → `proposed` → **черга керівнику** (затвердити/правити) → `approved`.
Зараз ролі per-компанія [Є], але потоку "підлеглий пропонує → керівник затверджує" нема —
додати як стан у контурі.

### 4.2. Тихий канал — новий, критичний
Interview-агент дає співробітнику позначити відповідь конфіденційною. Така відповідь:
- НЕ йде в чергу керівнику, НЕ пише в граф;
- створює `REDFLAG_RAISED` **тільки вам** (навіть власник не бачить);
- у панелі довіри — окремий розділ "тихі прапорці".
Технічно: окремий scope доступу, видимий лише super-admin. Без цього губиться ваша
ключова цінність (ловити "утаїли/замовчали").

### 4.3. Документи — розширити Document Reader
Читання завантажених файлів → витяг фактів у контур із провенансом "з документа". Частково
перетинається з наявним імпортом Excel/CSV [Є], але треба вміти неструктуровані
(регламенти, переписки), не тільки таблиці.

---

## 5. Розбіжність джерел — новий механізм

Зараз двоє джерел = друге перезаписує перше (або конфлікт непомітний). Стає (ваше рішення):
- Новий факт суперечить наявному від іншого джерела → НЕ перезапис, а `SOURCE_DISAGREEMENT`
  + обидві версії в `conflict_group` + статус поля `disputed`.
- Підсвітка вам: "власник X, команда (N) Y — розберись".
- Rule Engine знає: на `disputed` не будувати "здорову" оцінку.

---

## 6. Чеклист повноти — зробити мовчання видимим

Агент зараз веде вільний діалог по стадіях. Додати **coverage map**: для кожного
відділу/посади/процесу — перелік обов'язкових полів. Агент наполегливо добуває кожне
(кілька спроб, переформулювання); недобуте → "не покрито" в панелі. Ви бачите не "готово",
а "процес продажів: 8/12, відповідального за крок 5 ніхто не назвав". Це технічна основа
вашого "не забули/не утаїли".

---

## 7. Delivery Engine — новий модуль (ваш біль №2)

Наявні стадії онбордингу (A-H у візії) перетворити на явну машину станів:
- 6 етапів E1-E6 з **критеріями готовності = виконання правил Rule Engine + % покриття +
  рівень довіри** (Документ 1 §9). Етап не закривається формально, поки є діри.
- **Панель контролю (ваш головний екран):** кілька клієнтів, по кожному — етап, що блокує,
  конфлікти, тихі прапорці, хто гальмує (не відповів агенту), рівень довіри. Портфель
  клієнтів [Є] — це його розширення, не новий продукт.
- Delivery читає провенанс і Rule Engine — тому будується ПІСЛЯ розділів 1-3.

## 10. Google Drive — ваше рішення лишилось відкритим
Ви не відповіли на питання Drive (пропустили), тому фіксую як розвилку, не нав'язую.
Проблема реальна: токен протермінований [Част], критики попереджають про split-brain
(працівник править Google Doc, а в БД інший стан). Три варіанти — редактор усередину /
лишити Drive / гібрид (редактор усередині, на Drive експорт PDF). **Рекомендація:** для
вашої моделІ (ви контролюєте дані) гібрид найбезпечніший — правда в БД під провенансом,
Drive лише для фінальних PDF назовні. Але це рішення за вами; воно не блокує розділи 1-7.

## 11. Порядок робіт

| Фаза | Зміст | Чому в цьому порядку |
|---|---|---|
| Ф1 | Провенанс + статуси фактів (§1) | фундамент усього; без нього ваша роль неможлива |
| Ф2 | Rule Engine окремим шаром (§2) | читає граф, живить Delivery і панель |
| Ф3 | Event Bus + ідемпотентність (§3) | вмикає "реальний час" і фонового сторожа |
| Ф4 | Три канали: черга затвердження + тихий канал + документи (§4) | ваша ключова цінність |
| Ф5 | Розбіжність джерел + чеклист повноти (§5,§6) | робить контроль якості реальним |
| Ф6 | Delivery Engine + панель контролю (§7) | збирає все в робочий інструмент диригента |

Ф1-Ф3 — інфраструктурні, майже без UI. Ф4-Ф6 — де з'являється ваша щоденна цінність.
Drive (§10) — паралельно, коли вирішите.
