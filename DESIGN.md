# Жива Орг.Платформа — Технічний дизайн

> Платформа-«живий організм»: орг.структура + бізнес-процеси + посадові інструкції як єдина
> взаємопов'язана система. Мультитенант SaaS. Усі зміни — через спільний Telegram-бот із ШІ.
> Дата фіксації дизайну: 2026-07-08.

---

## 0. Концепція

Компанія (від 5 осіб) описує себе один раз через опитування в боті. ШІ генерує **чернетку**
орг.структури, процесів і посадових інструкцій. Далі система «жива»: кожна зміна інструкції
через вектор-пошук знаходить пов'язані інструкції, ШІ пропонує правки, відповідальний керівник
підтверджує їх у Telegram. Інструкції зберігаються як Google Docs на Диску клієнта (звично людям),
структура зв'язків і метадані — у нашій БД.

## 2. Зафіксовані рішення (узгоджено з власником)

1. **Джерело правди — гібрид.** Орг.структура + процеси = структуровано в Postgres. Тексти
   інструкцій = Google Docs на Drive клієнта. БД тримає зв'язки + метадані. Вектор-індекс з Drive.
2. **Розповсюдження змін = ШІ пропонує → людина підтверджує** (через `proposals` + inline-кнопки в Telegram).
3. **Вектор-база = pgvector** у нашому Postgres, ізоляція по `company_id`. У нас лише ембединги, тексти — на Drive.
4. **Ядро = форк fineko**, зверху шар вектор-пошуку + Drive-sync + API для бота.
5. **Онбординг = мердж**: діалогова логіка з воронки курсу + генерація структури з fineko.
6. **Результат генерації = чернетка (DRAFT)**, яку далі правлять.
7. **Ролі/доступ:** керівник відділу — правки в межах свого відділу; керівник компанії — по всій компанії; співробітник — свої інструкції.
8. **Один спільний бот платформи** для всіх компаній. Маршрутизація через deep-link коди.
9. **Мультитенант** закладаємо в модель даних із самого початку (`company_id` всюди — вже є у fineko).
10. **Мова — лише українська.**
11. **Підписка (місячна)** — слоти в схемі закладаємо зараз, білінг вмикаємо пізніше.
12. **Завантаження документів через бот:** користувач кидає файл → ШІ витягує текст, класифікує
    посаду/процес/папку → створює Google Doc у правильній теці Drive → створює запис instruction + зв'язки → індексує.

## 3. Стек

- **Мова:** Node.js + TypeScript (єдина з fineko і воронками).
- **API:** Express (з fineko backend_new).
- **Frontend:** Next.js + Tailwind + Radix (з fineko frontend).
- **БД:** PostgreSQL + Prisma + **pgvector**.
- **Черги/фон:** Bull (як у воронках) — для розповсюдження змін і переіндексації.
- **ШІ:** Claude (Anthropic) основний + OpenAI/Gemini fallback (пакет `claude` з воронок). Ембединги — Google Vertex `text-embedding-004` (логіка з ПСЗ) або OpenAI `text-embedding-3-small`.
- **Google Drive/Docs API** — створення тек і документів, читання для індексації.
- **Деплой:** VPS + PM2 + git-pull (модель воронок).

## 4. Модель даних (розширення схеми fineko)

Наявні моделі fineko лишаються. Додаємо/розширюємо:

- `instruction_links` — явний граф залежностей: `instruction_id`, `related_instruction_id`, `relation_type`, `weight`.
- `instruction_embeddings` (pgvector) — `company_id`, `instruction_id`, `chunk_no`, `content`, `embedding vector`, `drive_file_id`, `updated_at`.
- `drive_folders` — `company_id`, `path`, `drive_folder_id`, прив'язка до вузла орг.структури.
- `drive_files` — `company_id`, `drive_file_id`, `folder_path`, `instruction_id`, `mime`, `last_indexed`, `status` (аналог «Індексовані файли» ПСЗ).
- `proposals` (розширити) — `type` (instruction_edit / structure_change / new_doc), `payload(JSON)`, `status`, `target_instruction_id`, `created_by`, `approved_by`.
- `companies` (+поля) — `drive_root_folder_id`, `link_code`, `subscription_plan`, `subscription_status`, `trial_ends_at`.
- `company_members` (+поля) — `telegram_user_id`, скоуп по відділах (через наявну `department_company_member`).
- `bot_links` — `code`, `company_id`, `member_id`/`role`, `expires_at` — deep-link коди для спільного бота.

---

## 5. Маршрутизація спільного бота (deep-link)

Один бот на платформі. Підключення: `https://t.me/<bot>?start=<link_code>`.
На `/start <code>` бот резолвить компанію + учасника з `bot_links`, прив'язує `telegram_user_id`
до `company_member`. Далі всі повідомлення йдуть у контексті цієї компанії/ролі.
(Патерн з fineko `telegram_auth_codes` / `group_link_codes` + воронки `resolveTargetBot`.)

## 6. Ключові сценарії (flows)

1. **Онбординг-опитування** — бот-воронка веде діалог, на етапах б'є в API платформи. Результат: сирі відповіді про бізнес.
2. **Генерація чернетки** — ШІ будує орг.структуру (divisions→departments→sections→positions), процеси, посадові інструкції (status=DRAFT). На Drive створюються теки за орг.структурою (з шаблону клієнта), інструкції — Google Docs у правильних теках, усе індексується в pgvector.
3. **Зміна інструкції через бот** — правка тексту → оновлення Doc + ре-ембединг → вектор-пошук пов'язаних → ШІ формує `proposals` → відповідальний керівник підтверджує/відхиляє inline-кнопками → на підтвердження пов'язані Docs оновлюються + ре-ембедяться.
4. **Додавання документа через бот** — файл → витяг тексту → класифікація (вектор + LLM) → Google Doc у потрібній теці → instruction + зв'язки → індексація.
5. **Розповсюдження (worker)** — при будь-якій зміні фоновий Bull-джоб знаходить пов'язані/схожі інструкції, створює proposals, сповіщає відповідальних.

## 7. Архітектура сервісів

```
Платформа (цей репозиторій, VPS)
├── apps/api        ← Express API (онбординг, генерація, правки, аппрув, upload)
├── apps/worker     ← Bull: розповсюдження змін, переіндексація, сповіщення
├── apps/web        ← Next.js фронт (форк fineko frontend): орг.структура, процеси, інструкції
├── packages/db     ← Prisma + pgvector
├── packages/ai     ← Claude + ембединги (fallback OpenAI/Gemini)
├── packages/drive  ← Google Drive/Docs sync + індексація (порт ПСЗ)
└── packages/telegram ← хелпери сповіщень

Бот-воронка (система для воронок) — Telegram UX, викликає API платформи через HTTP/MCP.
```

Розподіл: **бот-воронка = обличчя/діалог**, **платформа = мозок + дані + Drive + вектори**.

## 7a. Фази (Sheets-first)

**Фаза 1 — Sheets-first MVP (без веб-інтерфейсу).**
- Джерело правди для орг.структури та процесів = **Google Sheet** (звичний ручний формат власника).
- Інструкції = Google Docs у теках Drive за орг.структурою.
- Наша БД тримає лише прив'язки: `companies`, `drive_folders`, `drive_files`, `instruction_links`, `instruction_embeddings`, `bot_links`, `proposals`. Зв'язок «інструкція ↔ посада ↔ процес» пишемо в БД одразу (щоб уникнути болючої двобічної синхронізації пізніше).
- ШІ: онбординг-опитування → генерація структури/процесів у Sheet + інструкцій у Docs → індексація в pgvector.
- Бот: правки інструкцій (proposals → аппрув), завантаження документів.

**Фаза 2 — Веб-інтерфейс (форк fineko).**
- Постгрес стає повноцінним джерелом правди для структури; Sheet — експорт/перегляд.
- Додаємо UI орг.структури, процесів, інструкцій, чергу proposals.

**Фаза 3 — Комерція.** Вмикаємо підписку/білінг (слоти в схемі вже є).
